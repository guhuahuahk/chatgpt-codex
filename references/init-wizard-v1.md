# 初始化向导 v1

这是当前版本的创建向导合同。

## 当前目标

只创建“瘦身版可运行 agent”，不在这一版里展开完整角色卡。

## 层级规则

- 默认由当前调用该 skill 的 agent 创建自己的直属下级
- 默认继承当前调用者所在部门
- 不默认跨部门创建
- 新的部门总经理或更高层岗位，应由更高层控制者发起

## 当前版最少问答

1. 新成员叫什么名字
2. 现在是否立刻接 bot
3. 如果要接 bot，再要 token

如果用户提供了 token，初始化向导不能在“已写入 token”处结束，必须继续把入口真正带到可用状态。

当前版不要主动追问：

- 详细角色定义
- 完整职责边界
- 输出物类型
- 复杂权限
- 复杂 heartbeat 策略

这些属于后续升级版向导。

## 当前版默认推断

- `agentId`：根据中文名转拼音 / ASCII / kebab-case
- 上级：当前调用者
- 部门：当前调用者所在部门
- 入口：默认不接外部 bot
- heartbeat：默认先明确策略；如无外部入口，可保留单一中文主会话；如有外部直聊入口且要求单窗口，优先把 heartbeat 绑定到真实直聊 session；若暂时不需要 heartbeat，再改为 `0m`
- 用户可见名称：严格使用用户提供的中文名，不拼内部 `agentId / accountId`

## 如果用户选择“现在接 bot”

初始化向导必须继续走完下面这些步骤：

1. 用 token 建立该 agent 专属的 channel/account 绑定
2. 重启或刷新 gateway，让 bot 真正上线
3. 明确提示用户：
   - 现在请给这个 bot 发一条消息
   - 这是为了激活第一条真实会话
4. 等待真实 direct 会话长出来
5. 检查 UI 是否出现重复窗口：
   - 常见情况是 `main/heartbeat` 窗口和真实私聊窗口同时存在
6. 如果出现重复窗口，执行会话收口：
   - 保留真实私聊窗口
   - 如果需要保留 heartbeat，把 heartbeat 绑定到这条真实会话的 `session key`
   - 如果暂时不需要 heartbeat，再将该 agent 的 heartbeat 调整为 `every: "0m"`
   - 备份冗余 heartbeat 或临时主会话
   - 从 live 会话索引中移除多余窗口
   - 同步清理数据库中的旧 heartbeat 会话记录
7. 最后再向用户汇报：
   - bot 已接好
   - 新窗口已激活
   - 会话已收口/已合并

也就是说，用户给完 token 后，这个向导还要继续负责“激活 + 收口”，不能中途停下。

## 当前版完成标准

完成后，至少要得到：

- 独立 workspace
- 独立 `agentDir`
- 独立 `sessions`
- 标准化主会话显示名，且用户可见名称只保留中文名
- `openclaw.json` 中的 `agents.list` 记录
- PostgreSQL 中的正式 agent 档案

如果用户在本次向导中提供了 token，还必须额外得到：

- 外部 bot 可用
- 第一条真实外部会话已生成
- 多余窗口已完成收口

## 当前版状态口径

只允许汇报：

- 已建立瘦身版可运行 agent
- 已建立瘦身版可运行 agent，但有缺口

不要把“只建骨架”说成“已建立 agent”。
